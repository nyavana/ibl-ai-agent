
## Frozen v1 spike storage choice

For `bwm_ephys` v1, the default spike storage choice is:

- encoding: `delta_int_ticks`
- quantization: `100 us`
- compression: `shuffle_zstd`

In short:

```text
delta_int_ticks + 100 us + shuffle_zstd
```

This decision is recorded in:

- `docs/decisions/bwm_ephys_spike_encoding.md`

## Scope

This document defines a Brain Wide Map dataset centered on good-unit
electrophysiology, with enough session, trial, event, spike, task-response, and
passive-response context to answer many reviewed questions from a single
self-contained release. Wheel and pose support live in the companion
`bwm_behavior` dataset rather than in `bwm_ephys`.

For broad questions about where task, movement, pose, or behavior-state
information is represented in the brain, a future compact neurobehavior feature
atlas may be a better query surface than the full `bwm_ephys` plus
`bwm_behavior` pair. That proposed layer would precompute per-unit PETH
features, behavior cross-correlations, QC, location, and ephys-atlas-style
features. See `docs/decisions/bwm_dataset_layering.md`.

Dataset identity:
- Name: `bwm_ephys`
- Current public version: `1.2.0`
- Initial target version: `1.0.0`

Pinned source scope (initial intent):
- Freeze: `2023_12_bwm_release`
- Sorter revision for good-unit selection: `2024-05-06`
- Good-unit rule: `label >= 1.0`

Core product definition:
- One downloadable release artifact for users
- Internally organized as multiple files/directories by access pattern
- Analysis-ready rather than raw-complete
- Ephys-centered, but includes behavioral and event context needed to interpret the neural data
- Does not claim to include all possible BWM modalities such as raw video

Recommended distribution artifact:
- `reports/releases/bwm_ephys/1.2.0/bwm_ephys-1.2.0.tar`

The latest `bwm_ephys` archive is generated by the passive upgrade path and
written to that release folder automatically. To write or refresh only the
tar/checksum for an already-built dataset without recomputing the dataset,
use:

```bash
UV_CACHE_DIR=.uv-cache uv run ibl-ai-agent write-bwm-release-archive --dataset-root <bwm_ephys-version-dir>
```

Meaning of `tar`:
- `tar`: archive format that bundles many files/directories into one archive
- Recommended over `.zip` for a large scientific dataset release when the
  payload is already compressed internally
- Used here as the portable release/handoff artifact, not as the main local
  query interface

## Design Goals

- One dataset download, not many coordinated downloads
- Self-contained for most reviewed questions
- Fast local loading with `pandas`, `polars`, `duckdb`, `xarray`, or custom Python code
- Stable join keys and explicit schema
- Avoid live Alyx/ONE access at analysis time
- Separate relational/tabular access from chunked time-series access
- Keep feature recomputation out of common analysis paths when possible
- Preserve room for future compression and quantization experiments without breaking the schema contract
- Make the dataset easy for an agent to query directly

## Naming Rationale

Preferred public dataset name:
- `bwm_ephys`

Why this name fits:
- The primary payload is electrophysiology-centered
- The dataset includes spikes, units, insertions, channels, and ephys-derived features
- Trials/events are included as context for ephys analyses; wheel/pose are handled by the companion `bwm_behavior` dataset
- The name does not overclaim modality completeness the way `full` might

## High-level Storage Strategy

Use different formats for different access patterns.

Implemented in the current local artifact:

- **Parquet** for metadata tables and feature tables
- **directory-based blosc shards** for spike arrays
- **YAML/JSON** for schema, manifest, provenance, and build reports

Future-compatible design targets:

- **Parquet** for metadata tables, feature tables, and aggregate summaries
- **Zarr** may still be considered for future large chunked numeric arrays such
  as wheel traces and pose traces, and possibly spikes
- **YAML/JSON** for schema, manifest, provenance, and build reports

Recommended principle:
- one release archive
- many internal files
- no single monolithic internal file

## Data Model and Join Keys

Primary keys:
- `sessions`: `eid`
- `insertions`: `pid`
- `units`: `(pid, cluster_id)`
- `channels`: `(pid, channel_id)`
- `trials`: `(eid, trial_id)`
- `events`: implementation-dependent; recommended `(eid, event_id)` or `(eid, trial_id, event_name)`

Core joins:
- `insertions.eid -> sessions.eid`
- `units.pid -> insertions.pid`
- `units.eid -> sessions.eid`
- `(channels.pid) -> insertions.pid`
- `(units.pid, units.channel_id) -> (channels.pid, channels.channel_id)` when channel assignment is present
- `trials.eid -> sessions.eid`
- `events.eid -> sessions.eid`
- `trial_features.(eid, trial_id) -> trials.(eid, trial_id)`
- `unit_features.(pid, cluster_id) -> units.(pid, cluster_id)`

Optional convenience key:
- `unit_id`: stable synthetic identifier derived from `(pid, cluster_id)`

## Spike Data Organization

### Core requirement

`bwm_ephys` should include all spikes from good units.

### Existing source organization to preserve

The current source organization is per insertion, with arrays like:
- `spikes.times`: sorted spike times
- `spikes.clusters`: cluster assignment for each spike

This organization should be preserved conceptually in the dataset design.

### Current storage format

The current implementation uses **directory-based blosc shards**, organized per
insertion.

Implemented location:
- `spikes/`

Implemented shard structure:

```text
spikes/
  <pid>/
    meta.json
    spike_times_delta_ticks.blosc
    spike_clusters.blosc
    cluster_ids.blosc
    cluster_spike_counts.blosc
```

Minimum arrays per insertion:
- `spike_times_delta_ticks`: delta-encoded quantized spike times for all spikes
  in the insertion
- `spike_clusters`: cluster index/cluster id for each spike
- `cluster_ids`: list of included good-unit cluster ids represented in this insertion payload
- `cluster_spike_counts`: per-cluster spike totals

Per-insertion metadata currently lives in `meta.json`, including fields such as:
- `eid`
- `pid`
- `probe_name`
- `subject`
- `date`
- `session_number`
- `lab`
- `time_quantization_us`
- `time_encoding`
- `cluster_encoding`
- `compression_config`
- `n_spikes`
- `n_good_units`

### Why preserve `(spike_times, spike_clusters)`

Pros:
- matches existing IBL/BWM mental model
- easy to understand
- supports reconstruction of unit-specific trains by filtering `spike_clusters`
- supports event-aligned analysis while preserving a familiar representation

Tradeoff:
- extracting one unit requires masking/filtering within the insertion payload unless extra indexing metadata is added

Optional indexing helpers:
- `unit_offsets` when the data is also grouped by unit internally
- `cluster_spike_counts`
- per-unit manifest rows in a small auxiliary table

For v1, preserving the familiar `(spike_times, spike_clusters)` concept remains
the guiding choice even though the implemented on-disk representation is
`spike_times_delta_ticks` plus metadata.

## Spike Time Quantization and Compression Policy

### Motivation

Spike times are likely to dominate storage size. Compression strategy should therefore be configurable and profileed.

### Configurability requirement

The schema and builder should support iterating on multiple quantization/compression settings before the release is finalized.

The chosen configuration must be recorded in:
- `provenance.yaml`
- `manifest.json`
- per-shard metadata in the spike store

### Recommended baseline approach

1. Start from sorted spike times per insertion
2. Quantize times to integer ticks
3. Preserve sorted order
4. Optionally apply delta encoding to improve compressibility
5. Store compressed arrays in the spike store

### Quantization unit

Recommended initial experimental target:
- `0.1 ms` (`100 us`) ticks

Important:
- Quantization must remain a release-level parameter
- Different quantization settings must not silently coexist within the same dataset version unless explicitly documented

Recommended schema field / attr:
- `time_quantization_us: int`

Example interpretations:
- `100` means one tick = `0.1 ms`
- `1000` means one tick = `1 ms`

### Integer representation

Recommended strategy:
- convert spike times to nonnegative integer ticks after quantization
- use `uint32` when safe for the full recording duration and tick size
- otherwise use `uint64`

The exact dtype should be recorded in the manifest/schema.

### Delta encoding

Delta encoding is strongly recommended as an experiment and may become the default if profiles are favorable.

Concept:
- absolute ticks: `t0, t1, t2, ...`
- delta ticks: `t0, dt1, dt2, dt3, ...` where `dt_i = t_i - t_{i-1}`

Why this helps:
- spike times are sorted
- delta values are much smaller than absolute times
- lower entropy usually compresses better under `zstd`

### Current implemented storage stack

The current local artifact uses explicit file shards rather than Zarr:

- container format: `blosc_file_shards`
- compression: blosc + zstd + shuffle
- stored arrays per insertion:
  - `spike_times_delta_ticks`
  - `spike_clusters`
  - `cluster_ids`
  - `cluster_spike_counts`

Zarr remains a possible future alternative, but it is not the current v1
implementation in this repo.

### Spike compression modes to support in the builder

The builder/spec should support at least these modes:
- `float_seconds`
- `int_ticks`
- `delta_int_ticks`

Where:
- `float_seconds`: raw float seconds, baseline mode
- `int_ticks`: quantized absolute integer ticks
- `delta_int_ticks`: quantized integer ticks stored as first absolute tick plus deltas, or fully delta-encoded stream depending on implementation

Recommended configuration fields:
- `time_encoding`
- `time_quantization_us`
- `time_dtype`
- `delta_encoded: bool`
- `compressor`
- `filters`

### Builder profiling requirement

Before freezing a public release, profile at least:
- final file size
- decode/read performance
- event-alignment performance for common reviewed workflows
- fidelity impact of the chosen quantization

Recommended profile matrix:
- `0.1 ms` ticks, absolute integer
- `0.1 ms` ticks, delta integer
- `1 ms` ticks, absolute integer
- `1 ms` ticks, delta integer
- raw float baseline

The chosen release setting should be based on these profiles, not hard-coded in advance.

## Column Encoding and Data Types

Recommended type policies:
- repeated low-cardinality strings: dictionary-encoded in Parquet where supported
- counts: `int32` unless range requires `int64`
- time columns in metadata tables: `float32` seconds unless higher precision is essential
- coordinates: `float32`
- booleans: true boolean type
- dates: `YYYY-MM-DD` string for portability

Region naming policy:
- keep Allen-derived acronym fields
- keep Beryl-derived acronym fields
- optionally include Cosmos mapping fields
- document all derivations explicitly in `schema.yaml` and `provenance.yaml`

## Sorting, Partitioning, and Chunking

### Parquet tables

Recommended policy:
- avoid over-partitioning into many tiny files
- prefer a small number of deterministic files per logical table
- sort rows deterministically by useful join keys

Suggested sort keys:
- `sessions`: `eid`
- `insertions`: `pid`
- `units`: `pid`, `cluster_id`
- `channels`: `pid`, `channel_id`
- `trials`: `eid`, `trial_id`

### Array stores

Current implementation:
- spikes sharded naturally by insertion

Future design target:
- wheel chunked naturally by session
- pose chunked naturally by session

This matches expected access patterns and simplifies partial rebuilds.

## Release Variants and Scope Management

The `bwm_ephys` name should denote a self-contained dataset with metadata tables included.

Recommended interpretation:
- `bwm_ephys` contains both metadata/tables and ephys payloads
- users should not need a separate metadata package just to use the dataset

Optional future family structure if needed:
- `bwm_simple`: deprecated lightweight metadata-only legacy dataset
- `bwm_ephys`: preferred metadata + spikes + ephys features dataset for new work
- `bwm_behavior`: preferred companion dataset for wheel, pose, and trial-behavior features

## How `bwm_ephys` Helps Reviewed Questions

This organization directly simplifies reviewed analyses by turning many current data-loading problems into local table queries or direct access to aligned spike arrays.

Examples:

### Good units per insertion by lab
Use:
- `metadata/insertions.parquet`
- or `metadata/units.parquet`

### Firing rate vs spike width by region
Use:
- `metadata/units.parquet`
- `features/unit_features.parquet`

### Total spikes across BWM
Use:
- `metadata/units.parquet[spike_count]`
- or `features/insertion_features.parquet`

### Latency to peak response after stimulus onset
Use either:
- current `features/event_response_features.parquet` when the stored event/window definition matches the question, or
- current `spikes/<pid>/...` shard payloads + `metadata/trials.parquet`

### Wheel- or pose-linked neural analyses
Use:
- current `spikes/<pid>/...` shards
- `metadata/trials.parquet`
- companion `bwm_behavior` wheel/pose/trial-behavior feature tables
- explicit joins through stable keys such as `eid`, `trial_id`, `pid`, and `cluster_id`

- `metadata/sessions.parquet`
- `metadata/insertions.parquet`
- `metadata/units.parquet`
- `metadata/channels.parquet`
- `metadata/trials.parquet`
- `metadata/events.parquet`
- `features/unit_features.parquet` when feasible
- `spikes/` shard store
- `manifest.json`
- `schema.yaml`
- `provenance.yaml`
- `SUMMARY.md`

Strongly recommended:
- `features/insertion_features.parquet`
- `wheel/by_session.zarr`
- `features/trial_features.parquet`

Optional in later iterations:
- `dlc/by_session.zarr`
- `features/psth_features.parquet`
- aggregate summary tables

## Open Decisions for the Builder

These are intentionally left configurable and should be implemented as explicit build parameters:
- spike time quantization unit
- absolute vs delta time encoding
- integer dtype for encoded spike times
- Zarr codec/filter stack
- inclusion scope for pose channels/signals
- whether to add unit-level indexing helpers beyond `spike_times` + `spike_clusters`
- whether to include precomputed PSTH/modulation features in v1 or later

The builder should record all chosen values in release metadata so a dataset version is fully self-describing.
